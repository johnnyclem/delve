# Bestiary monster artwork

The Bestiary panel renders pixel-art portraits next to every monster
card and a hero image at the top of each detail view. Images are
generated by an offline script and stored in object storage; the
database keeps only the URL + provenance.

## How it works

- Schema: `monster_images` (lib/db/src/schema/monster_images.ts) —
  one row per SRD slug, shared across the 2014 + 2024 editions.
- Storage: Replit object storage, served via the existing
  `/api/storage/objects/<id>` route (auth-gated by campaign
  membership, same as character portraits).
- Generation: `pnpm --filter @workspace/api-server run gen:bestiary-images`
  walks distinct monster slugs in `reference_chunks`, skips slugs
  already in `monster_images`, calls `gpt-image-1` via the existing
  AI integration proxy with a fixed style prompt, uploads the PNG,
  and inserts the row.

## Running the script

From the repo root:

```bash
# Generate every missing slug (~330 monsters per edition, ~330 distinct
# slugs total since editions share slugs).
pnpm --filter @workspace/api-server run gen:bestiary-images

# Top-up only a handful (useful for smoke-testing after a SRD re-import).
BESTIARY_IMAGE_LIMIT=10 pnpm --filter @workspace/api-server run gen:bestiary-images

# Print prompts without calling OpenAI or writing to storage.
BESTIARY_IMAGE_DRY_RUN=1 pnpm --filter @workspace/api-server run gen:bestiary-images
```

The script is idempotent and resumable — re-running only generates
slugs that don't already have a row, so partial runs (e.g. interrupted
by a transient OpenAI 5xx) safely continue on the next invocation.

Env knobs:

- `BESTIARY_IMAGE_LIMIT=N` — cap the number of slugs processed this run.
- `BESTIARY_IMAGE_CONCURRENCY=N` — parallel OpenAI calls (default `2`).
- `BESTIARY_IMAGE_DRY_RUN=1` — log prompts only.

## Cost

`gpt-image-1` at `1024x1024` standard quality is roughly $0.04 per
image. ~330 distinct slugs ≈ **~$13** for a full one-shot run. Re-runs
are free for slugs that already exist; only newly-added SRD entries
incur cost.

> The task plan originally specified `512x512` rendering, but
> `gpt-image-1` only supports `1024x1024`, `1024x1536`, and
> `1536x1024` (the smaller sizes are dall-e-2 only). We render at
> `1024x1024` and rely on the FE `image-rendering: pixelated` plus the
> 64-80px thumbnail downscale to preserve the chunky pixel-art look.

## Running once against production

After deploying schema changes (`drizzle-kit push` against the prod
DB), run the script once with `DATABASE_URL` pointed at production
(use the Object Storage tool pane for the prod bucket — the env vars
`PRIVATE_OBJECT_DIR` and `DEFAULT_OBJECT_STORAGE_BUCKET_ID` must match
the prod bucket). This is the same pattern as the existing
`seed:test-user` prod run documented in TEST_ACCOUNT.md.

## Changing the art style

The style prompt lives in **one constant** (`STYLE_PROMPT` in
`scripts/gen-bestiary-images.ts`) and is saved alongside each row in
`monster_images.prompt`. To re-style the whole bestiary:

1. Edit `STYLE_PROMPT`.
2. `TRUNCATE monster_images;` (or delete the slugs you want to
   regenerate).
3. Re-run the script — costs match the table above.

The DB-side `prompt` column lets you diff the live prompt against the
script's current `STYLE_PROMPT` before deciding to wipe.
